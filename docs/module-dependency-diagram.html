mlf4 / mlf2 Module Dependency Diagram

This view compresses the Haskell module graph into dependency domains derived from imports in `src/`, `src-public/`, `src-research/`, `app/`, and `test/`. Numbers on arrows are cross-domain import counts, so the diagram shows which module families depend on which others most heavily.

Public entry structure

  • • `MLF.API` is mostly a frontend façade: 7 imports, all into `MLF.Frontend.*`.
  • • `MLF.Pipeline` is the real bridge: it touches frontend, one constraint module, and one elaboration module.
  • • `MLF.Program` is thin compatibility glue over `MLF.API` + `MLF.Pipeline`.
  • • `MLF.XMLF` is a wrapper over explicit xMLF syntax/parse/pretty helpers.

Heavy dependency hubs

  • • `MLF.Constraint.*` is the largest domain at 82 modules.
  • • `MLF.Elab.*` depends most heavily on `Constraint` (70 cross-domain imports).
  • • `Constraint` itself leans hardest on `Util` (36 imports) and lightly on `Binding` / `Reify`.
  • • `Reify` forms a feedback edge back into `Constraint` (20 imports), reflecting graph-to-type reconstruction.

Non-production consumers

  • • `test/Main.hs` and its spec tree are the widest consumers of internal modules.
  • • `src-research` stays separate, but still depends on frontend/constraint/elab families.
  • • `app/Main.hs` is lightweight: it mostly delegates into `MLF.Program.CLI` and research entrypoints.
  • • This diagram is intentionally domain-compressed, not a full per-module graph.
