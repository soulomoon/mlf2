# Goal Mechanism Table (Current vs Target)

Last updated (UTC): 2026-05-02
Goal: Keep xMLF as the thesis-faithful typed elaboration IR while making `MLF.Backend.IR` the single executable eager backend IR, avoiding a duplicate public `LowerableBackend.IR` unless a later evidence-backed boundary requires it.
Source of truth: `papers/these-finale-english.txt`; `docs/architecture.md`; `src/MLF/Backend/IR.hs`; 2026-05-02 backend IR design discussion.

| Mechanism | Current codebase behavior | Target behavior | Gap summary | Evidence (spec/code/tests) | Gate (YES/NO) | Next action |
|---|---|---|---|---|---|---|
| IR role separation and non-duplication | xMLF/`ElabTerm` and `MLF.Backend.IR` both exist, and architecture docs describe the backend boundary, but the "one backend IR, no duplicate public ANF IR" goal is not yet encoded as a durable invariant. | xMLF remains the typed/theory elaboration representation; `MLF.Backend.IR` is the only public backend IR; any ANF-like normalization stays internal unless a documented boundary earns its own invariants. | The distinction is understood in discussion but not yet enforced in docs, tests, or module contracts. | `docs/architecture.md` backend boundary section; `src/MLF/Backend/IR.hs` module note. | NO | Update architecture guidance and backend module notes to state the one-backend-IR contract and the criteria for adding any later lower IR. |
| Eager runtime lowering contract | `BackendExpr` is typed and includes lambdas, applications, lets, ADTs, cases, type abstraction/application, roll/unroll, and closure nodes. | `Backend.IR` states the eager runtime/codegen contract explicitly without importing lazy STG machinery such as thunks, update frames, CAF update semantics, or graph reduction. | The IR is backend-owned, but the eager-runtime contract is partly implicit and split between conversion, validation, and LLVM lowering. | `BackendClosure`, `BackendClosureCall`, `BackendConstruct`, `BackendCase`; `docs/architecture.md` closure ABI section. | NO | Document and test which runtime concerns belong in `Backend.IR`, which belong in LLVM lowering, and which STG-like lazy concerns are intentionally out of scope. |
| Direct calls, closure values, and callable shapes | Direct applications use `BackendApp`; indirect closure calls use `BackendClosureCall`; validator rejects some confused shapes. | Callable shape should be explicit enough that direct calls, local function calls, and closure calls are hard to confuse and diagnostics identify the violated backend invariant. | The current split is useful but still relies on validation and lowerer conventions instead of a more precise callable representation. | `BackendClosureCalledWithBackendApp`; closure validation errors in `MLF.Backend.IR`. | NO | Decide whether to refine callable forms inside `Backend.IR` or keep the current shape with stronger validation tests and clearer diagnostics. |
| ADT/case semantics versus layout | `BackendData`, `BackendConstructor`, `BackendConstruct`, and `BackendCase` preserve semantic ADT/case structure for lowering. | Semantic constructor/case structure stays explicit, while representation decisions such as tag values, field layout, boxed/unboxed field treatment, and nullary strategy are made in a documented backend-owned place. | Layout policy exists mostly as lowerer behavior rather than an explicit contract adjacent to the IR. | Constructor metadata in `MLF.Backend.IR`; lowering-owned constructor runtime metadata in `MLF.Backend.LLVM.Lower`. | NO | Add a documented layout boundary or evidence-backed tests that lock the current lowerer-owned layout decisions. |
| Primitive operations and eager evaluation order | Many operations can appear as ordinary applications or backend-supported patterns; expression evaluation order is mostly lowerer-owned. | Primitive operations and effect-sensitive evaluation order are explicit enough for eager LLVM lowering, either in `Backend.IR` itself or in a private normalizer inside the LLVM backend. | As supported programs grow, implicit evaluation order and primitive lowering rules may become fragile. | Current expression-shaped `BackendExpr`; LLVM lowering diagnostics for unsupported expressions/calls. | NO | Identify current primitive/evaluation-order assumptions and decide the narrowest explicit representation needed before adding more effects, FFI, or primitive ops. |
| Polymorphism erasure and lowerability | `BackendTyAbs` and `BackendTyApp` may remain in `Backend.IR`; lowerers reject unsupported polymorphic shapes or specialize selected paths. | The backend contract clearly says which polymorphic nodes are still allowed at `Backend.IR` and what must be erased, specialized, or rejected before LLVM emission. | Lowerability is partly a property of the LLVM path rather than an explicit named backend invariant. | `BackendTyAbs`, `BackendTyApp`; backend conversion and LLVM unsupported-node diagnostics. | NO | Define and test the supported polymorphism-erasure/specialization boundary for LLVM-lowerable backend programs. |
| Validation, evidence, and guidance synchronization | `validateBackendProgram` checks local invariants; docs describe current backend behavior; tests cover selected conversion/lowering cases. | Every target backend invariant above has focused tests or documentation evidence, and guidance stays synchronized across `AGENTS.md`, `docs/architecture.md`, and module notes when contracts change. | Some goals are currently design intent rather than verified mechanism. | `validateBackendProgram`; `docs/architecture.md`; backend tests referenced by cabal/test stanzas. | NO | Add or update focused tests and docs with each mechanism improvement; run the narrowest relevant test slice before the full gate for behavior changes. |

Rules:
- Keep mechanism order fixed.
- Use only `YES` or `NO`.
- Flip `NO -> YES` only with passing verification evidence.
